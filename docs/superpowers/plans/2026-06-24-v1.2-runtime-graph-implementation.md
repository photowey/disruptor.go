# V1.2 Runtime Graph Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans or superpowers:subagent-driven-development to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Objective:** Ship `v1.2.0` graph semantics and runtime routing while
preserving the existing V1 publish path and V1.1 static graph hot path.

**Architecture:** Static `Graph[T]` keeps one processor per real handler node and treats `START`/`END` as export and validation terminals only. `RuntimeGraph[T]` is a separate API that runs one scheduler processor per event, evaluates conditional routing edges, executes selected real nodes, and advances producer gating once the runtime event reaches `END` or an explicit no-route policy completes it.

**Tech Stack:** Go generics, `ringbuffer.RingBuffer`,
`processor.BatchEventProcessor`, `ringbuffer.Barrier`, `sequence.Sequence`,
standard `testing`, expression parser implemented in `pkg/expression`, existing
metrics and examples layout.

---

- Create `pkg/runtimegraph/graph.go`: runtime graph builder, options, validation, snapshots, and export.
- Create `pkg/disruptor/runtime_graph_processors.go`: runtime graph handler, handle options, no-route policy, metrics, and exception handling.
- Create `pkg/runtimegraph/graph_test.go`, `pkg/expression/expression_test.go`, and `pkg/disruptor/runtime_graph_test.go`: runtime graph and expression coverage.

- Modify `pkg/disruptor/disruptor.go`: allow `HandleRuntimeGraph` as graph mode and reject mode mixing.

- Modify examples, `README.md`, `README.zh-CN.md`, `docs/api-guide.md`, and benchmarks.
